# shipyard-cp v0.1.0

`shipyard-cp v0.1.0` は、複数の AI provider / worker を有限ネストで上流オーケストレーションする control plane の初回リリースです。

今回のリリースでは、backend / worker / CLI を主導線、Web UI を補助導線とする形に整理し、acceptance を含む主要フローが実運用可能な水準まで揃いました。

## ハイライト

- CLI-first の運用導線を整備
- acceptance 完了フローを backend / UI の両方で閉じられるように修正
- frontend の build / test blocker を解消
- task / run / timeline / audit の補助UIを、誤認を減らす方向に整理
- README、CLI ハブ、Skills、RUNBOOK を更新して入口情報を一本化

## 何ができるか

- task を `plan -> dev -> acceptance -> integrate -> publish` の段階で管理できる
- Codex / Claude Code / Google Antigravity / GLM-5 系 worker を共通の task / run 契約で扱える
- task / run / timeline / audit を通じて、進行状況と結果を追跡できる
- CLI から日常運用しつつ、必要時だけ Web UI で補助確認できる

## このリリースで整えたこと

### Backend / Worker

- acceptance worker が `accept` verdict を返した後、hidden step で止まらず terminal state へ進めるよう修正
- audit summary と timeline の整合性を改善
- build / test を通した状態に整理

### Frontend

- dashboard / task detail / run detail / settings まわりの主要 UX 不具合を修正
- acceptance 完了 API を UI から呼べる導線を追加
- TypeScript build blocker、テスト不安定要因、モック残骸を整理
- 補助UIとしての命名、情報密度、導線を改善

### Docs / Operations

- README を CLI-first / finite nesting / orchestration の本質に合わせて全面整理
- `docs/cli-usage.md` を追加し、入口情報を一本化
- repo-local Skills を追加
- `.claude/commands/README.md` を追加し、`run / status / pipeline` の違いを明確化
- root 直下の運用不要 docs を `docs/` 配下へ整理
- infra 資材を `infra/` に集約

## 検証状況

リリース前に以下を確認済みです。

- root: `pnpm test`
- root: `pnpm run build`
- web: `npm test`
- web: `npm run build`
- Playwright による補助UIの実操作確認
- API / worker 主体フローの acceptance 完了確認

## 既知の前提

- Web UI は主役ではなく補助UIです
- API は internal contract として残していますが、人の通常運用は CLI-first を前提としています
- 今回は product を優先しており、docs の継続改善余地は残っています
